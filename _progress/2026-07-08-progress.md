
## New skills batch + real email attachments (2026-07-08)
Shipped:
1. **Real email attachments** — lib/email.ts loadAttachments() reads a file,
   base64-encodes it, caps total at 10MB; Mailjet Base64Content + nodemailer
   Buffer both wired. email.send primitive gained attach_paths (comma/JSON/
   array, path-safety gated same as fs.read_external). POST /api/email/send
   also accepts attachPaths. Verified live: Summit Recon.xlsx (610KB) sent
   as a real attachment via Mailjet.
2. **4 new skills**: send-attachment, human-handoff (human.request), 
   payment-collection (Paynow/Stripe), dataset-publish (Intellinexus) — all
   previously-shipped features with zero skill coverage. Registered in
   SKILL_KEYWORDS with regex triggers.
3. **Found + fixed the ROOT CAUSE of the original Summit Recon incident**:
   defaultVaultPlan's needle-extraction regex ran its "and <verb> ...$"
   tail-strip AFTER a broader leading-verb strip that already deleted the
   trigger word (summarize/attach/etc) — so the tail-strip silently no-op'd,
   corrupting the fs.find_in needle into "Summit Recon and it in an email to
   X" (exactly what the incident logs showed). Fixed by reordering: tail-strip
   now runs FIRST. Also added a dedicated attach-and-send branch: when a task
   says "find X ... attach ... email" with a literal address present, the
   plan now chains fs.find_in → email.send(attach_paths) instead of stopping
   after find. Named recipients (no literal address) degrade safely to
   plain find (no guessed address).
4. PLAN_SYSTEM (LLM planner) got a matching ATTACHMENTS rule + example.
5. email-writing's SKILL_KEYWORDS trigger narrowed (negative lookahead
   excluding "attach") so it stops tying with send-attachment and losing —
   ties were resolved by alphabetical order, silently picking the wrong skill.

77/77 tests (13 new), typecheck clean, live-verified end-to-end via the exact
