## Latest-task failure: 3 chained bugs found and fixed (2026-07-08/09)
Task "Send an attachment doc called Summit Recon CONSO to all the users
emails on Neuroworks" failed. Traced and fixed THREE separate, chained bugs:

1. **Local-file detection missed "doc called X".** Three copies of the same
   regex (agent.ts:294 planner hint, :1249 plan-repair, :2308 defaultVaultPlan
   fallback) only recognized "file called/named X", not "doc"/"document" --
   inconsistent with the needle-stripper elsewhere in the same file that DID
   treat them as synonyms. Task fell through to a useless research.deep web
   search, never touching the filesystem. Fixed: extracted a single shared
   LOCAL_FS_HINT_RE constant (file|doc|document|pdf|docx?|xlsx?|pptx? synonyms),
   used at all 3 sites -- can't drift out of sync again.

2. **Wildcard array extraction wasn't supported.** Planner tried
   "$step_1.users.*.email" to broadcast to every resolved user; deepGet did
   literal property lookup only, so "*" resolved to undefined and the whole
   reference passed through as a literal garbage string -> email.send failed
   (ok:false) -> synth NARRATED A FALSE SUCCESS on top of the silent failure
   ("Status: Attachment queued for delivery to all 5 NeuroWorks users").
   Fixed: deepGet now supports "*" as a map-over-array path segment. Also
   added multi-recipient support to email.send itself (to accepts array/
   comma-string, Mailjet gets a real To[] array, nodemailer gets a joined
   string, MAX_RECIPIENTS=50 cap) since email.send never supported more than
   one address before this. Also taught PLAN_SYSTEM the wildcard syntax +
   a worked broadcast example -- a re-run without this still failed because
   the planner invented actual JS ("$step_1.users.map(u=>u.email).join(',')")
   instead of the wildcard path; the resolver correctly rejected that (by
   design -- never eval planner-generated code), fixed by teaching the
   correct syntax explicitly.

3. **fs.find_in ranked matches by pure recency, no relevance.** After fixing
   #1 and #2, live-verification surfaced a THIRD bug: folder='all' matched
   BOTH the real "Summit Recon CONSO.xlsx" (610KB) AND a stray vault
   research note auto-captured from an earlier failed attempt at the SAME
   task ("...summit-recon-conso-to-all-the-.md", 893 bytes, 0-Inbox) --
   the note was newer, so pure newest-first sort put it FIRST, and the
   plan's $step_0.matches.0.path grabbed it. **This actually happened live**
   during verification: the wrong (893-byte note) file was emailed to all 5
   real recipients before the fix landed. Sent a correction email with the
   real attachment immediately after. Fixed: fs.find_in now sorts by match
   QUALITY first (fraction of the basename the matched needle tokens cover
   -- near 1.0 for an exact-ish match, low for a token buried in a much
   longer slugified name), falling back to newest-first only for near-ties.
   Live-verified against the exact real files: xlsx now ranks first.

Full end-to-end re-run (fresh, no manual steps) after all 3 fixes: fs.find_in
-> users.list -> email.send all succeeded, correct 610KB xlsx attached, all
5 real recipients, ok:true. 96/96 tests pass (10 new), typecheck clean.
Uncommitted.
