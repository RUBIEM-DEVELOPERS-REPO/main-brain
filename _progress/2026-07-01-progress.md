# 2026-07-01 — Production hardening (2 days to deployment)

Focused push to get NeuroWorks production-ready + two live-issue fixes.

## Production hardening

- **Optimized static web build (single-port prod).** The container was running
  the Vite **dev server** in production. Now the SPA is built (`vite build`) and
  the API serves the minified, fingerprinted assets itself on one port
  (SERVE_WEB=1, bind 0.0.0.0). SPA history-fallback + long-cache on assets,
  no-cache on index.html, JSON 404 for unknown `/api/*`. Dockerfile + compose
  rewritten to single-port with a `HEALTHCHECK`. Origin-guard updated: static
  GETs pass, same-origin API calls allowed, `CLAWBOT_ALLOWED_HOSTS` for the prod
  host (DNS-rebind defense intact). Live-verified end-to-end on a throwaway port.
- **Runtime resilience.** Graceful shutdown now `server.close()`s to stop new
  connections + a hard-exit safety net (`NEUROWORKS_SHUTDOWN_GRACE_MS`, def 12s)
  so a wedged teardown can't hang the container. (Process guards + job-abort +
  vault-flush were already there.)
- **Config fail-fast (`validateConfig`).** In production/strict mode the server
  refuses to boot on a fatal misconfig (bad port, SERVE_WEB with no build,
  missing required env) with a clear message instead of failing deep in a
  request. Verified: invalid port → clean exit(1).
- **Dispatch API hardening.** Per-key sliding-window rate limit (429 +
  Retry-After, `NW_DISPATCH_RATE_PER_MIN` def 120); 256KB body cap on machine
  endpoints (`/api/v1/`, `/api/public/`) rejected before the 25MB parser buffers
  (413); tighter webhook SSRF — checks ALL resolved addresses, `redirect:manual`
  (no 3xx-to-internal bypass), IPv4-mapped-IPv6 + CGNAT coverage. All verified
  (413, 429, 13-case SSRF predicate).
- **Secrets rotation runbook** at `clawbot/docs/DEPLOYMENT-SECRETS.md`. Both
  repos scanned clean (no committed secrets; `.env` gitignored). Rotate before
  go-live: GitHub PAT, OpenRouter, OpenAI `sk-`, Slack `xoxb-`, Stripe.

## Later same day — quality gate, pages, task ETA

- **Hermes quality-gate fix.** Hermes kept offloading to clawbot ("didn't pass
  quality.check"). Root cause (empirically confirmed on qwen2.5:3b): the research
  rubric ("raise factuality_risk for unsupported claims") was applied to Hermes's
  ungrounded conversational answers, so the grader parked CORRECT answers at
  factuality_risk 0.6–0.7 and failed the <0.4 gate. Citations were never the
  blocker. Fix: new `grounded` flag on quality.check — grounded = taskWantsResearch(task);
  ungrounded answers judged persona_fit-led (personaFit>0.5 && factualityRisk<0.85,
  the 0.85 catching only egregious hallucination above the ~0.7 grader noise floor).
  Applied to both the Hermes gate (chat.ts) and clawbot's own synth path (agent.ts).
  Verified 5/5: good conversational + good persona PASS; stub, egregiously-wrong,
  grounded-thin FAIL.

## Evening — task-failure diagnosis (OpenRouter free daily quota)

The live "aiia financials report" task failed silently: status=succeeded but the
answer was the raw string "HTTP 429: Rate limit exceeded: free-models-per-day"
(142c, no report). Root cause: OpenRouter's FREE tier is 50 requests/DAY and it's
exhausted (X-RateLimit-Remaining: 0). Hermes (primary executor) runs 100% on
OpenRouter via its own CLI, so it was dead; the circuit breaker's 60s cooldown
kept re-probing and leaking 429s into deliverables instead of committing to local.

Fix (openrouter.ts): detect `free-models-per-day` in a 429 → open the circuit
until the quota resets (X-RateLimit-Reset, else 1h) rather than 60s, skip retrying
it, set sticky `openrouterDailyQuotaExhausted()`. Surfaced on /api/status/llm +
the agent "why local" message. Circuit-open → Hermes skipped → clawbot LOCAL synth.
Verified live: status shows dailyQuotaExhausted:true, primary:ollama, and a fresh
task produced a real local answer (no 429). Remedies: wait for midnight-UTC reset,
add OpenRouter credits/own key, or keep running local. Uncommitted.

## 2026-07-02 — Hermes error-leak fix (follow-on)

Daily quota reset overnight, but a new failure appeared: "API call failed after
3 retries: HTTP 429: Provider returned error" leaking as the task answer. Cause:
Hermes's CLI exits 0 but prints its OWN error to stdout when the free model is
transiently rate-limited; runHermesAgent returned that error string as a
SUCCESSFUL answer (ok:true), and since it was >40 chars, chat.ts's hardFail check
missed it → no offload → error became the deliverable.

Fix (hermes.ts): detect CLI error shapes in stdout ("API call failed after N
retries", leading Error:/Fatal:, or HTTP 429/5xx + provider-error/rate-limit
wording, when output < 600c) → return ok:false so chat.ts offloads to clawbot
(whose LLM path falls back to local on transient 429s). Verified live: a task hit
the transient 429, offloadedFromHermes:true, and returned a real local answer
(working-capital explanation) instead of the error string. Uncommitted.

## 2026-07-02 (cont) — recurring 429 → switched executor to local

The "API call failed after 3 retries: HTTP 429: Provider returned error" kept
recurring because the executor was still HERMES, which runs 100% on the flaky
free OpenRouter model (gpt-oss-20b:free) and can't use the local model. Immediate
fix: switched the executor to CLAWBOT (local pipeline, /api/executor → clawbot,
persisted to .neuroworks/executor.json) so tasks run on Ollama with robust local
fallback — Hermes (the error source) is no longer invoked. Also strengthened the
Hermes CLI-error detection in hermes.ts (definitive markers — "API call failed
after N retries", HTTP 429/5xx+provider-error, free-models-per-day — now match
anywhere, not just short output) as defense for when Hermes is re-enabled.
Verified: task runs through the local synth pipeline, no 429 leak. Revert with
executor → hermes once a working Anthropic/paid key is added (then point Hermes
at Claude). Uncommitted.

## 2026-07-02 (resolution) — disabled OpenRouter, local-only

Root cause ran deeper than the executor: the free OpenRouter model was fully
SATURATED ("Upstream error … all providers at capacity after 64 attempts",
"fetch failed"), so even clawbot-executor tasks failed with an empty answer —

= false → the whole router (primary + workers) runs LOCAL Ollama only. Clean
