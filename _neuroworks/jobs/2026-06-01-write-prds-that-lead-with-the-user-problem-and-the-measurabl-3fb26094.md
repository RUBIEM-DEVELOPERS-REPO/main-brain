## Product Requirements Document (PRD) – Problem‑First, Outcome‑First Template  

| Section | What to Include | Why it Matters (tied to the user problem & measurable outcome) |
|---------|----------------|---------------------------------------------------------------|
| **1. Title & Owner** | • Concise, descriptive name  <br>• Product manager, team, date | Provides a clear reference point for every stakeholder. |
| **2. Problem Statement** | • **Who** – the target user persona  <br>• **What** – specific pain point (quote or metric)  <br>• **Why** – impact on workflow, revenue, churn, etc. | Starts the PRD with the *user problem* rather than a feature idea, keeping the team focused on solving a real need. |
| **3. Success Metrics (Measurable Outcome)** | • Primary KPI(s) (e.g., % reduction in task time, NPS lift, conversion increase)  <br>• Target value & timeframe (e.g., 30 % faster onboarding within 3 months)  <br>• Secondary metrics (support tickets, adoption rate) | Links the solution directly to a **quantifiable result**; the PRD is judged by hitting these numbers, not by delivering a feature list. |
| **4. Solution Overview** | • High‑level description of the proposed solution (not a detailed spec)  <br>• How it addresses the problem and drives the success metric | Gives engineers and designers context while maintaining outcome focus. |
| **5. Scope** | • **In‑Scope** – core capabilities required to achieve the metric  <br>• **Out‑of‑Scope** – optional or future work | Clarifies boundaries, preventing scope creep and aligning effort with the measurable outcome. |
| **6. Requirements** | • Functional requirements that directly support the solution overview  <br>• Non‑functional requirements (performance, security, compliance) | Ensures every requirement can be traced back to the problem and the success metrics. |
| **7. Risks & Mitigations** | • Identify assumptions, dependencies, and potential blockers  <br>• Planned mitigation actions | Proactively manages uncertainty that could impede achieving the outcome. |
| **8. Timeline & Milestones** | • Key delivery dates, MVP definition, release plan | Aligns team cadence with the timeframe specified in the success metrics. |
| **9. Stakeholders & Communication Plan** | • List of owners, reviewers, and communication channels | Guarantees transparency and timely feedback throughout the project lifecycle. |

### How to Use This Template
1. **Begin with the user problem** – capture the persona, pain, and business impact before any solution ideas.  
2. **Define measurable outcomes** – select clear, time‑bound KPIs that will prove the problem is solved.  
3. **Iterate the solution only as a high‑level concept** that directly maps to the metrics.  
4. **Validate each requirement** against the problem statement and success metrics to maintain focus.
