## Answer
**Sketching low‑cognitive‑load flows – a quick guide**

| Goal | Design tactic | Why it helps (cognitive‑load perspective) |
|------|---------------|-------------------------------------------|
| **Reduce the number of decisions a user must make** | **Group related actions into a single step** (e.g., “Add → Save” becomes “Add & Save”). | Fewer choice points lower *intrinsic* load and keep the user’s mental model simple. |
| | **Use progressive disclosure** – show only the next required option, hide the rest until it’s needed. | Limits *extraneous* load by preventing irrelevant information from competing for attention. |
| | **Provide clear defaults** that match the most common user intent. | Users can accept the default with a single tap, avoiding unnecessary deliberation. |
| **Make the flow easy to scan and remember** | **Apply a consistent visual hierarchy** (size, color, spacing) for primary vs. secondary actions. | Consistency creates *schema* shortcuts, so users recognise patterns without re‑learning each step. |
| | **Label actions with verb‑first, outcome‑focused text** (e.g., “Send invitation” vs. “Invite”). | Action‑outcome pairing reduces the mental translation step, cutting *germane* load. |
| **Guide the user’s attention** | **Use directional cues** (arrows, progress bars, step numbers). | External cues act as scaffolding, keeping the user’s focus on the current step. |
| | **Limit on‑screen elements to 3‑5 per view** (Miller’s “7 ± 2” rule). | Prevents overload of working memory, making each decision feel manageable. |
| **Support error recovery** | **Show inline validation** immediately after an input, with concise corrective suggestions. | Early feedback prevents the accumulation of errors, reducing the need for costly mental back‑tracking. |
| | **Offer an “undo” or “back” option that returns to the exact previous state**. | Gives users confidence to explore options without fear of losing progress, lowering anxiety‑related load. |
| **Test and iterate** | **Run quick low‑fidelity usability tests** (paper sketches, click‑through prototypes). | Direct observation reveals hidden decision points and unnecessary steps that inflate cognitive load. |

### How to apply these tactics in a sketch

1. **Start with a high‑level map** – draw the main goal (e.g., “Create event”) as a single node.  
2. **Branch out only when a decision is truly required** – each branch should represent a *must‑choose* option, not an optional tweak.  
3. **Add visual cues** (step numbers, arrows) directly on the sketch to indicate the intended sequence.  
4. **Annotate defaults and hidden steps** (e.g., “auto‑fill date = today”).  
5. **Review the sketch**: count the decision nodes. Aim for ≤ 3 per screen; if more, look for ways to combine or defer them.
