## Answer
## Trigger  
The runbook is initiated when the **NeuroWorks platform fails to start** after a system reboot or when the service manager reports the `neuroworks` service as **inactive** or **failed**.

## Preconditions  
1. Administrator has SSH access to the host running NeuroWorks.  
2. The host OS is Ubuntu 22.04 LTS (or compatible Debian‑based distribution).  
3. The `clawbot` user has sudo privileges without a password prompt.  
4. No other maintenance tasks are scheduled for the same host during execution.  
5. Sufficient disk space (≥ 5 GB) is available on the root filesystem.  

## Steps  

1. **Connect to the host**  
   ```bash
   ssh clawbot@<host‑address>
   ```

2. **Check the service status**  
   ```bash
   sudo systemctl status neuroworks.service
   ```

3. **Inspect the latest log entries**  
   ```bash
   sudo journalctl -u neuroworks.service -n 20 --no-pager
   ```

4. **Verify that the required configuration file exists and is readable**  
   ```bash
   sudo test -r /etc/neuroworks/config.yaml && echo "Config present"
   ```

5. **Confirm that the Python virtual environment is intact**  
   ```bash
   source /opt/neuroworks/venv/bin/activate
   python -c "import neuroworks; print(neuroworks.__version__)"
   deactivate
   ```

6. **Ensure the data directory permissions are correct**  
   ```bash
   sudo chown -R clawbot:clawbot /var/lib/neuroworks
   sudo chmod -R 750 /var/lib/neuroworks
   ```

7. **Attempt to start the service**  
   ```bash
   sudo systemctl start neuroworks.service
   ```

8. **Wait 10 seconds, then re‑check the status**  
   ```bash
   sleep 10
   sudo systemctl is-active neuroworks.service
   ```

9. **If the service is active, clear the stale lock file** (if it exists)  
   ```bash
   sudo rm -f /var/run/neuroworks.lock
   ```

10. **Record the outcome in the operations log**  
    ```bash
    echo "$(date '+%Y-%m-%d %H:%M:%S') - NeuroWorks start attempt: $(systemctl is-active neuroworks.service)" >> /var/log/neuroworks/ops.log
    ```

## Verification  
- `systemctl is-active neuroworks.service` returns **active**.  
- `curl -s http://localhost:8080/health` returns `{"status":"ok"}`.  
- No error lines appear in `sudo journalctl -u neuroworks.service -n 10`.  

## Rollback  

1. **Stop the service**  
   ```bash
   sudo systemctl stop neuroworks.service
   ```

2. **Restore the previous configuration backup** (if a backup exists at `/etc/neuroworks/config.yaml.bak`)  
   ```bash
   sudo cp /etc/neuroworks/config.yaml.bak /etc/neuroworks/config.yaml
   ```

3. **Revert permissions on the data directory to the state before step 6** (use the snapshot recorded in `/var/log/neuroworks/perm_snapshot.log` if available).  
   ```bash
   # Example restoration command
   sudo chown -R root:root /var/lib/neuroworks
   sudo chmod -R 755 /var/lib/neuroworks
   ```

4. **Restart the service to apply the reverted state**  
   ```bash
   sudo systemctl start neuroworks.service
   ```

5. **Confirm rollback success** by repeating the **Verification** steps; the service should be inactive or report the original error condition.
