## Answer
**How to Write Release‑Notes That Lead With the User‑Facing Benefit**

| Step | What to do | Why it matters (user benefit) |
|------|------------|------------------------------|
| 1️⃣ Identify the top‑level value | Start each note with the *outcome* the user will see (e.g., “Faster project loading saves you 30 seconds per session”). | Users instantly understand *what’s in it for them* without parsing technical details. |
| 2️⃣ Follow with a brief “what’s changed” line | After the benefit, add a one‑sentence description of the change (e.g., “We optimized the file‑indexing algorithm”). | Keeps the note concise while still giving context. |
| 3️⃣ Add optional technical details | If needed, list a bullet‑point of deeper technical info for power users or support staff. | Provides depth for those who need it without cluttering the headline. |
| 4️⃣ Use consistent structure | **Benefit → Change summary → Technical details** (if any). | Predictable format lets readers scan quickly across multiple releases. |
| 5️⃣ Write in plain language | Avoid jargon; use verbs that describe user impact (“reduces,” “enables,” “prevents”). | Improves comprehension for non‑technical audiences. |
| 6️⃣ Highlight any user‑action required | If the update needs a setting change or migration step, place that at the end with a clear call‑to‑action. | Prevents confusion and reduces support tickets. |
| 7️⃣ Test the headline | Read the first line aloud; ask a colleague: “If you only saw this, would you know why you should care?” | Validates that the benefit is truly front‑and‑center. |

### Example (before vs. after)

**Before**  
> *Added support for JSON import.*  
> Fixed a bug that caused crashes when opening large files.

**After (benefit‑first)**  
> **Import data in seconds:** JSON files now load up to 5× faster, letting you start analysis sooner.  
> • Added JSON import support with optimized parsing.  
> • Fixed crash on large‑file opening.
