# Research: As a Product Designer, critique designs against the user's job-to-be-done — not personal taste

**How a Product Designer Should Critique a Design – Focus on the User’s Job‑to‑Be‑Done (JTBD), Not Personal Taste**

| Step | What to Do | Why it Matters (JTBD Lens) | How to Phrase the Feedback |
|------|------------|---------------------------|----------------------------|
| 1️⃣ Define the JTBD | Start every critique by restating the specific job the user is trying to accomplish (e.g., “Help a busy professional file expense reports in ≤ 2 minutes”). | Keeps the conversation anchored to the outcome the user actually needs, not to the designer’s aesthetic preferences. | *“The core JTBD is X. Does this element help the user achieve X within the required constraints?”* |
| 2️⃣ Map design elements to JTBD steps | Break the JTBD into sub‑tasks (discover, act, verify, etc.) and check which UI components support each step. | Shows whether the design is *functional* for the job, revealing gaps or redundancies. | *“The ‘quick‑add’ button supports the ‘capture expense’ step, but the confirmation modal adds friction to the ‘verify’ step.”* |
| 3️⃣ Measure against success criteria | Use the JTBD’s success metrics (time, error rate, satisfaction) as the benchmark for critique. | Objective criteria replace subjective taste; you can point to data‑driven gaps. | *“Target is ≤ 2 seconds to add an item; the current flow averages 4 seconds, so we need to streamline the input fields.”* |
| 4️⃣ Identify friction points | Highlight any UI/UX detail that slows, confuses, or adds risk to completing the job. | Friction directly reduces the user’s ability to get the job done efficiently. | *“The dropdown list contains 30 options, causing decision‑fatigue; consider auto‑complete or grouping.”* |
| 5️⃣ Propose JTBD‑aligned alternatives | Offer concrete, job‑focused revisions rather than “I like/dislike” statements. | Alternatives are judged by how well they improve job performance, not by personal style. | *“Replace the multi‑step wizard with an inline edit that lets users add details without leaving the list, cutting the task time in half.”* |
| 6️⃣ Validate with users | Recommend quick usability tests that measure the JTBD metrics after changes. | Empirical validation ensures the critique leads to real improvements. | *“Run a 5‑minute task test with 8 users and record time‑to‑completion; compare against the 2‑minute target.”* |

### Why This Approach Works
- **Objective grounding** – By tying every comment to the JTBD, feedback becomes measurable rather than opinion‑based.  
- **User‑first mindset** – Designers stay aligned with what the user actually needs to achieve, preventing “designer‑centric” solutions.  
- **Actionable outcomes** – Each critique ends with a concrete, job‑focused recommendation that can be tested.

### Quick Checklist for a JTBD‑Focused Review
- ☐ Does the design enable the user to start the job quickly?  
- ☐ Are all required steps supported without unnecessary clicks or screens?  
- ☐ Does each interaction reduce error risk or cognitive load?  
- ☐ Are success metrics (time, accuracy, satisfaction) likely to improve?  
- ☐ Can the proposed changes be validated with a short user test?
