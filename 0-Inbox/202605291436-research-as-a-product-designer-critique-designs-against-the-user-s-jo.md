# Research: As a Product Designer, critique designs against the user's job-to-be-done — not personal taste

**How a Product Designer Should Critique Designs – Focus on the User’s Job‑to‑Be‑Done, Not Personal Taste**

| Step | What to Do | Why it Matters for the JTBD |
|------|------------|----------------------------|
| **1. Identify the core job** | Pull the explicit *job‑to‑be‑done* (JTBD) statement from the user research vault (e.g., “help me quickly compare pricing options while staying within my budget”). | The JTBD defines the *outcome* the user is trying to achieve; every design decision should be measured against how well it supports that outcome. |
| **2. Map the design to JTBD outcomes** | List the key outcome metrics embedded in the JTBD (speed, accuracy, confidence, effort, error‑reduction, etc.). For each screen or interaction, note which outcome it influences. | This creates a direct traceability matrix: *design element → desired user outcome*. It prevents subjective “I like the colour” comments. |
| **3. Use evidence‑based criteria** | • **Functional fit** – Does the element enable the user to complete the job? <br>• **Efficiency** – Does it reduce the steps or time needed? <br>• **Clarity** – Does it make the required information obvious? <br>• **Error tolerance** – Does it prevent or recover from mistakes? | These criteria are grounded in the JTBD’s success metrics (e.g., “complete the task in < 2 minutes”). |
| **4. Quantify the impact** | Where possible, reference existing usability data, A/B test results, or heuristic scores that relate to the JTBD outcomes. If the vault contains prior research, cite the specific study or metric. | Quantitative evidence keeps the critique objective and tied to real user performance. |
| **5. Frame feedback as “job impact”** | Instead of “I don’t like this colour,” say “The current colour contrast increases visual search time, which adds ~3 seconds to the core task of locating the ‘Add to Cart’ button, reducing overall efficiency.” | The feedback directly connects a design choice to the user’s goal, making it actionable for the team. |
| **6. Prioritise recommendations** | Rank issues by the magnitude of their negative effect on the JTBD (e.g., high‑impact friction points first). Use a simple scale: *Critical – Blocks the job*, *Major – Significantly slows it*, *Minor – Small annoyance*. | Prioritisation aligns the design roadmap with the most valuable user outcomes. |
| **7. Validate with the user** | After revising, run a quick usability test focused on the JTBD success criteria (time‑on‑task, error rate, satisfaction). Record whether the changes move the metrics in the desired direction. | Closing the loop ensures the critique leads to measurable improvement, not just aesthetic preference. |

### Quick Checklist for a JTBD‑Centric Critique

- [ ] Have I restated the user’s JTBD in plain language?  
- [ ] Does each design element map to a specific JTBD outcome?  
- [ ] Is my feedback expressed in terms of *impact on the job* (speed, accuracy, confidence, etc.)?  
- [ ] Have I backed the critique with data from the vault or prior tests?  
- [ ] Are recommendations ranked by their effect on the JTBD?  

---

**Bottom line:**  
A product designer’s critique should be a *job‑focused audit* rather than a personal style review. By anchoring every observation to the user’s defined job‑to‑be‑done and supporting it with evidence from the vault, feedback becomes objective, actionable, and directly tied to the product’s success.
