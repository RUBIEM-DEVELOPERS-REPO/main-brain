# Research: procurement request template for software acquisition business goal alignment cost of ownership onboarding training implementation options comparison

**Software‑Acquisition Procurement Request Template**  
*(Designed to keep the purchase aligned with business goals, show total cost of ownership, and enable clear comparison of implementation options.)*  

| Section | What to Include | Why it Matters (Evidence) |
|---|---|---|
| **1. Request Summary** | • Title of software (e.g., “Customer‑Insight Analytics Platform”)  <br>• Requestor name, department, and contact  <br>• Date of request | Provides a quick reference for all stakeholders. |
| **2. Business Goal Alignment** | • Specific strategic objective(s) the software supports (e.g., “Increase cross‑sell revenue by 8 % YoY”)  <br>• Key performance indicators (KPIs) to be impacted  <br>• Link to existing business case or roadmap | Procurement must be tied to the organization’s strategic needs, not just a “nice‑to‑have” tool【1†L1-L4】. |
| **3. Requirement Specification** | • Functional requirements (features, integrations, user limits)  <br>• Non‑functional requirements (security, compliance, scalability)  <br>• Preferred delivery model (SaaS, on‑prem, hybrid) | Clear specs are the first step in the procurement process and guide supplier proposals【1†L13-L16】. |
| **4. Cost‑of‑Ownership (TCO) Breakdown** | • License/subscription fees (annual or per‑user)  <br>• Implementation services (setup, data migration)  <br>• Onboarding & training costs  <br>• Ongoing support & maintenance  <br>• Potential hidden costs (e.g., extra storage, API calls) | Total cost of ownership is essential for comparing options and justifying spend【1†L13-L16】. |
| **5. Implementation Options & Comparison** | | |
|  5.1 Option A – Vendor‑Managed | – Scope of services (configuration, data migration)  <br>– Timeline (weeks)  <br>– TCO (sum of line items)  <br>– Risks (vendor lock‑in) | Shows a “turn‑key” path; useful when internal resources are limited. |
|  5.2 Option B – Internal Build‑Out | – Required internal staff (hours)  <br>– Estimated timeline  <br>– TCO (staff cost + tools)  <br>– Risks (skill gaps) | Allows control but may increase internal effort. |
|  5.3 Option C – Hybrid (Vendor + Internal) | – Split responsibilities  <br>– Timeline  <br>– TCO  <br>– Risks | Balances speed with internal knowledge transfer. |
| **6. Supplier Evaluation Criteria** | • Price (total cost)  <br>• Fit to functional & non‑functional specs  <br>• Implementation timeline  <br>• Vendor reputation & support SLA  <br>• Compliance (e.g., GDPR) | Procurement decisions weigh cost **and** other factors; the lowest bidder isn’t automatically chosen【1†L31-L34】. |
| **7. Risk & Mitigation** | • Key risks (e.g., data migration failure, integration delays)  <br>• Mitigation actions (pilot, phased rollout) | Demonstrates strategic foresight and aligns with the broader procurement risk‑management process. |
| **8. Approval Workflow** | • Sign‑off matrix (requestor → department head → finance → procurement)  <br>• Required documentation (budget code, business case) | Procurement often involves multiple departments; a defined workflow ensures compliance【1†L13-L16】. |
| **9. Timeline & Milestones** | • RFP release date  <br>• Bid evaluation period  <br>• Contract award  <br>• Implementation start / go‑live | Keeps the project on schedule and informs all parties of expectations. |
| **10. Attachments** | • Detailed business case  <br>• Current system landscape diagram  <br>• Budget approval memo | Provides supporting evidence for reviewers. |

### How to Use the Template
1. **Fill out Sections 1‑3** to establish *why* the software is needed and *what* it must do.  
2. **Populate the TCO table (Section 4)** using vendor quotes and internal cost estimates.  
3. **Develop the three implementation scenarios (Section 5)** to enable side‑by‑side comparison.  
4. **Score each supplier against the criteria in Section 6**; the highest‑scoring option (not necessarily the cheapest) should be recommended.  
5. **Route the completed request through the approval workflow (Section 8)** before issuing an RFP or negotiating a contract.

This structure follows the strategic procurement steps—specifying requirements, evaluating alternatives, and aligning spend with business objectives—outlined in the definition of procurement as a “strategic process” rather than a purely transactional purchase【1†L1-L4】【1†L13-L16】.
