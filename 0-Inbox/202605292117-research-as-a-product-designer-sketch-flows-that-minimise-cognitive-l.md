# Research: As a Product Designer, sketch flows that minimise cognitive load and decision points.

Context: Role: Custom
Title: Sketch flows that minimise cognitive load and decision points
Description: Persona-derived starter task for "Dani"

**Sketch‑flow guidelines that keep cognitive load low and cut decision points**

| Goal | How to achieve it in the sketch flow | Why it matters (evidence) |
|------|--------------------------------------|---------------------------|
| **Limit the number of choices at each step** | • Use *progressive disclosure*: show only the next essential action, hide optional features behind a “More” or secondary screen.<br>• Group related actions into a single “primary” button and place secondary actions in a less‑prominent area (e.g., a bottom‑sheet or overflow menu). | A product is “something that is made to be **seen or experienced**, created to meet the needs or desires of consumers” — the experience should therefore be *simple* and focused on the core need, not cluttered with unnecessary options【1】. |
| **Create a clear visual hierarchy** | • Use size, colour, and placement to signal the *primary* path.<br>• Keep the number of visual elements per screen to a handful (3‑5) to avoid overload. | Cognitive load rises when users must parse many elements; a clean hierarchy reduces the “result of many hours spent” mental effort that users would otherwise expend【3】. |
| **Leverage familiar patterns** | • Adopt well‑known UI conventions (e.g., hamburger menu for navigation, thumb‑reachable tap targets).<br>• Re‑use the same interaction pattern for similar tasks throughout the flow. | Consistency aligns with the product’s “unique value proposition” and “quality” – users can rely on known behaviours, lowering the mental work needed to understand each step【1】. |
| **Provide immediate feedback** | • Show a brief confirmation (e.g., check‑mark, toast) after each action so users know they’re on the right track.<br>• Use progressive indicators (step numbers, loading bars) to convey progress. | Feedback turns the experience into a *result* that users can see, reinforcing that the product “sells itself” when it works smoothly【2】. |
| **Reduce the need for memory** | • Pre‑fill fields with known data or sensible defaults.<br>• Offer inline hints rather than separate help screens. | When the product “meets a real need” it should also *reduce* the effort required to achieve that need, otherwise the perceived value drops【1】. |
| **Keep the flow linear unless branching is essential** | • Map the user journey as a single path; only introduce branches when a clear, distinct user goal exists (e.g., “Save for later”).<br>• Label branches with explicit outcomes (“Save & Continue”, “Skip”). | A linear flow minimizes decision points, keeping the user’s mental model simple and preventing “decision fatigue”. |
| **Test with low‑fidelity sketches** | • Conduct quick paper‑prototype sessions to spot unnecessary steps.<br>• Iterate based on observed hesitation or back‑tracking. | Early validation ensures the flow truly “fulfills a real need” without hidden complexity【1】. |

### Quick sketch‑flow template (low‑fidelity)

1. **Welcome / Goal statement** – single headline, brief benefit.  
2. **Primary input** – one field or selection, auto‑filled where possible.  
3. **Confirm / Continue** – prominent CTA, secondary “Skip” link (low visual weight).  
4. **Feedback** – instant success toast + progress indicator (e.g., “Step 2 of 3”).  
5. **Next primary step** – repeat pattern; only introduce a branch if the user explicitly opts in.
