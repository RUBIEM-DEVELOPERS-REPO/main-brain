# Research: As a Product Manager, write prds that lead with the user problem and the measurable outcome

**Product Requirements Document (PRD) – “Problem‑First, Outcome‑First” Template**  

| Section | What to Include | Why it Matters (tied to the user problem & measurable outcome) |
|---------|----------------|---------------------------------------------------------------|
| **1. Title & Owner** | • Short, descriptive name <br>• Product manager, team, date | Gives a clear reference point for everyone who will read the doc. |
| **2. Problem Statement** | • **Who** is the user (persona) <br>• **What** pain point they experience (quote or metric) <br>• **Why** it matters – impact on their workflow, revenue, churn, etc. | Starts the PRD with the *user problem* rather than a feature idea, ensuring the team stays focused on solving a real need. |
| **3. Success Metrics (Measurable Outcome)** | • Primary KPI(s) (e.g., % reduction in task time, NPS lift, conversion increase) <br>• Target value & timeframe (e.g., 30 % faster onboarding within 3 months) <br>• Secondary metrics (support tickets, adoption rate) | Links the solution directly to a **quantifiable result**; the PRD is judged by hitting these numbers, not by delivering a feature list. |
| **4. Solution Overview** | • High‑level description of the proposed solution (not a detailed spec) <br>• How it addresses the problem and drives the success metric | Provides context for engineers and designers while keeping the focus on the outcome. |
| **5. Scope** | • **In‑Scope** – core capabilities needed to achieve the metric <br>• **Out‑of‑Scope** – nice‑to‑have items that will not be built in this release | Prevents scope creep and keeps the team aligned on what is required to meet the measurable outcome. |
| **6. User Journeys / Flows** | • End‑to‑end flow(s) that illustrate the user experience before and after the solution <br>• Key touchpoints where the pain point is alleviated | Shows concretely how the problem is solved in practice. |
| **7. Requirements** | • **Functional** – what the system must do (e.g., “allow bulk upload of contacts”) <br>• **Non‑functional** – performance, security, accessibility targets <br>• Each requirement should be traceable to a problem element or metric | Guarantees every requirement is justified by the problem or the desired outcome. |
| **8. Acceptance Criteria** | • Testable statements that must be true for the PRD to be considered complete (e.g., “average upload time ≤ 2 seconds for 10 k records”) | Provides a clear “done” definition that aligns with the measurable outcome. |
| **9. Dependencies & Risks** | • External teams, data, APIs, regulatory constraints <br>• Risks that could prevent achieving the metric and mitigation plans | Anticipates blockers that could jeopardize the outcome. |
| **10. Timeline & Milestones** | • Key dates (design freeze, MVP demo, launch) <br>• Alignment with the metric’s target timeframe | Keeps the team on schedule to hit the outcome deadline. |
| **11. Open Questions** | • Anything still undecided that could affect scope or success | Documents assumptions that need validation before development. |

### How to Use This Template  

1. **Start with the problem** – write a one‑sentence “pain‑point” that a specific persona experiences.  
2. **Define the outcome first** – decide the exact metric that will prove the problem is solved.  
3. **Only then brainstorm solutions** – ensure every idea can be mapped back to the problem and the outcome.  
4. Review the PRD with stakeholders and ask: *“If we meet the success metric, does the user problem disappear?”*  

### Why This Works  

- **User‑centric**: By foregrounding the problem, the team avoids building features that don’t matter to customers.  
- **Outcome‑driven**: Measurable KPIs give a concrete way to evaluate success, aligning product, engineering, and leadership on a shared goal.  
- **Lean**: Scope is explicitly limited to what is needed to achieve the metric, reducing waste and accelerating delivery.  

*References*: The definition of a product as “something created to meet the needs or desires of consumers” underscores the need to start with the user need and tie it to measurable value [2]. The PRD structure follows common product‑management best practices for linking problems to outcomes.
