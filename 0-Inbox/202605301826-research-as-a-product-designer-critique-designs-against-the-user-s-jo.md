# Research: As a Product Designer, critique designs against the user's job-to-be-done — not personal taste

**How a Product Designer Should Critique a Design – Focus on the User’s Job‑to‑Be‑Done, Not Personal Taste**

| Step | What to Ask | Why It Matters (tied to the user’s JTBD) |
|------|-------------|------------------------------------------|
| **1. Identify the Core Job** | *What fundamental problem is the user trying to solve?*  <br>‑ Look for the **core benefit** (e.g., “quench thirst” in a drink) rather than surface features. | Aligns the critique with the **core benefit** level of a product, the deepest user need [1]. |
| **2. Map the Desired Outcome** | *What specific result does the user expect after using the product?*  <br>‑ This is the **expected product** – the functional attributes the user assumes will be present. | If a design fails to deliver any expected outcome, it creates friction regardless of how “pretty” it looks. |
| **3. Evaluate Supporting Enhancements** | *What augmented features (extra value, convenience, delight) are promised?* | These are the **augmented product** elements that can differentiate a solution, but they must still serve the core job. |
| **4. Test Against Real‑World Context** | *Does the design work in the user’s actual workflow, environment, and constraints?* | A design that looks great on a mock‑up may break the **basic product** (the tangible artifact) when used in context. |
| **5. Measure Success Criteria** | *How will we know the job is done better?*  <br>‑ Define quantitative or qualitative metrics (time saved, error reduction, satisfaction). | Provides an objective yardstick that removes personal preference from the evaluation. |
| **6. Provide Actionable Feedback** | *What concrete changes will bring the design closer to the JTBD?* | Keeps the critique constructive and focused on solving the user’s problem, not on style. |

### Practical Tips for Staying JTBD‑Centric  

1. **Use the “Jobs Canvas”** – list the job, pains, gains, and success metrics before looking at the visual design.  
2. **Quote the user** – bring actual user statements into the critique to ground the discussion in real needs.  
3. **Prioritize impact over aesthetics** – rank feedback by how much it moves the design toward fulfilling the core job.  
4. **Avoid “I like/don’t like”** – replace taste‑based comments with “This element prevents the user from completing X” or “This addition speeds up Y by Z%.”

### Bottom Line  

A designer’s critique should be a **job‑focused audit**: confirm that the design delivers the core benefit, meets all expected outcomes, and adds meaningful augmentation without compromising the basic product. When feedback is tied directly to how well the design helps the user accomplish their job, personal taste becomes irrelevant.
