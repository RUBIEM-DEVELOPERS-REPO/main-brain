# Research: As a Technical Writer, write release notes that lead with user-facing benefit

**Release‑Notes Template – Lead with the User‑Facing Benefit**

| Section | What to Write | Why It Works (User‑Facing Focus) |
|---------|----------------|-----------------------------------|
| **Headline / Summary** | **Start with the primary benefit** – e.g., “*Faster project‑sync now lets you see teammates’ updates in real time*.” | Users read the first line; a clear benefit tells them *what’s in it for them* before any technical detail. |
| **Key Benefits (bulleted)** | • What problem is solved?<br>• How does it improve the user’s workflow?<br>• Any measurable gain (time saved, fewer clicks, higher accuracy)? | Bullets make the value instantly scannable; each point answers “Why should I care?” |
| **What’s New** | List the new features, UI changes, or performance improvements **in plain language**. Keep each item to one sentence and tie it back to the benefit when possible. | Users can quickly map the change to the benefit they just read. |
| **How It Works (optional)** | Brief, non‑technical description of the underlying change (e.g., “We introduced a background‑sync service that polls every 2 seconds”). | Provides context for power users without overwhelming casual readers. |
| **Impact / Migration** | • Any actions required?<br>• Compatibility notes?<br>• Rollback instructions (if needed). | Ensures users know what they need to do to enjoy the benefit. |
| **Known Issues** | List any remaining bugs that might affect the benefit. | Transparency builds trust. |
| **Links & Resources** | • Documentation URL<br>• Video demo<br>• Support contact | Gives users easy paths to explore the benefit further. |

### How to Write It (step‑by‑step)

1. **Identify the top user outcome** of the release (speed, simplicity, reliability, etc.).  
2. **Craft a one‑sentence benefit headline** that states that outcome directly.  
3. **Support the headline with 2‑4 benefit bullets** that quantify or qualify the improvement.  
4. **Add the technical details** only after the benefit section, using plain language.  
5. **Finish with actionable items** (migration steps, known issues, resources).

> *Example*  
> **Headline:** “*Instant task‑board refresh cuts wait time from 30 seconds to 2 seconds.*”  
> **Benefits:**  
> • See teammates’ changes instantly, keeping projects on track.  
> • Reduce idle time, boosting daily productivity by up to 15 %.  
> **What’s New:** Added a WebSocket‑based sync engine…
