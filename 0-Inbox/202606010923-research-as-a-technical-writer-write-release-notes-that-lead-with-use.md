# Research: As a Technical Writer, write release notes that lead with user-facing benefit

**Release‑Notes Template – Lead with the User‑Facing Benefit**

> *“What’s in it for me?”* – start every note with the concrete value the user will see.

---

### 1️⃣ Headline (Benefit‑First)

- **Format:** *Verb + Outcome + Feature*  
  Example: **“Speed up data imports by 40 % with the new bulk‑loader.”**  
- **Why:** The headline tells the user the direct advantage before any technical detail, matching the research focus on “lead with user‑facing benefit” [0‑Inbox/202605290928‑research‑as‑a‑technical‑writer‑write‑release‑notes‑that‑lead‑with‑use.md]​.

### 2️⃣ Short Summary (One‑sentence benefit expansion)

- Restate the headline in plain language, adding the context the user needs to understand the impact.  
  Example: “Large CSV files now load in a fraction of the time, letting you get to analysis faster.”

### 3️⃣ What’s New (Feature list)

| Feature | How it works (technical) | When it’s available |
|---------|--------------------------|---------------------|
| Bulk‑loader UI | New dialog with drag‑and‑drop, progress bar, and error‑summary panel. | v2.3 (June 2026) |
| Parallel processing engine | Uses all CPU cores automatically. | Same release |
| Configurable batch size | Set via Settings → Performance. | Same release |

> **Tip:** Keep each bullet to one line; the benefit stays front‑and‑center, the technical detail follows.

### 4️⃣ Why It Matters (Benefit reinforcement)

- Quantify the gain (e.g., “40 % faster”, “up to 2 × more rows per minute”).  
- Link to user scenarios (e.g., “Importing quarterly sales data no longer blocks other work”).

### 5️⃣ How to Get It

- **Upgrade path:** “Update via the in‑app updater or download the installer from the portal.”  
- **Prerequisites:** “Requires Windows 10 + 4 GB RAM.”  

### 6️⃣ Known Issues / Work‑arounds (Optional)

- List only issues that could affect the advertised benefit, with brief mitigations.

### 7️⃣ Additional Resources

- Link to the full changelog, documentation, and a short “Getting Started” video.

---

#### Quick Checklist for Each Release Note

- ☐ **Benefit first** – headline tells the user the outcome.  
- ☐ **Plain language** – avoid jargon in the first two sections.  
- ☐ **Technical details** – follow the benefit, not the other way around.  
- ☐ **Quantify** – include numbers (speed, time saved, accuracy).  
- ☐ **Actionable** – clear upgrade steps and any required settings.  

Using this structure ensures every release note immediately answers the user’s core question—*what’s in it for me?*—while still providing the necessary technical information for implementation.
